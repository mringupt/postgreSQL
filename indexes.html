JSONB @>, arrays, full-text tsvector, trigrams (pg_trgm). Multi-valued attributes.

Writes buffer to a pending list, merge on VACUUM. Why GIN queries sometimes spike.

🎯When to Use GiST
Range types, PostGIS geometry, exclusion constraints, nearest-neighbor search.
🔍Lossy Storage & RecheckImportant
Stores bounding box approximations. May return false positives that are rechecked.
🚫Exclusion ConstraintsUnique
Prevent overlapping ranges — only possible with GiST. Used in booking systems.
⚖️GiST vs GIN
GiST: faster writes, lossy. GIN: faster reads, exact. For FTS: GIN on static, GiST on frequently updated.

Query skips 512 pages (4MB) instantly. Only reads 384 pages of actual data. BRIN index size: ~2KB vs B-Tree equivalent: ~200MB
🎯When to Use BRIN
Append-only time-series, sequential IDs, naturally-ordered data. Correlation near 1.0.
📊Correlation CheckCritical
SELECT correlation FROM pg_stats — must be near ±1.0 for BRIN to work.
⚙️pages_per_range Tuning
Default 128. Smaller = more precise pruning, bigger index. Larger = less precise.
🪶Tiny Index SizeKey benefit
100M-row BRIN: ~50KB. Same B-Tree: ~2GB. Fits entirely in shared_buffers.

⚠️Equality OnlyLimitation
Only = operator. No ranges, no ORDER BY, no LIKE. Never range scan.
⚖️Hash vs B-Tree
O(1) theory vs O(log n) B-Tree. In practice: B-Tree height is 3–4. Difference negligible.
🤔When to Actually Use It
Rarely. Only if equality lookups on very long keys (UUIDs, long strings) and you've benchmarked it.
📜WAL HistoryInterview
Pre-PG10: not WAL-logged → not crash-safe. PG10+: fixed. Still rarely chosen.

❌ The Problem — Index Bypassed
-- Mixed-case stored data: INSERT INTO users (email) VALUES ('alice@example.com'), -- lowercase ('BOB@EXAMPLE.COM'), -- uppercase ('Charlie@Mix.Com'); -- mixed -- Plain B-Tree index: CREATE INDEX idx_email ON users (email); -- SLOW: lower() bypasses the index! SELECT * FROM users WHERE lower(email) = 'alice@example.com'; -- → Seq Scan (index ignored) 🐢
✅ Fix 1 — Expression Index
-- Index the expression, not the column: CREATE INDEX idx_email_lower ON users (lower(email)); -- Now this uses the index: SELECT * FROM users WHERE lower(email) = lower('Alice@Example.com'); -- → Index Scan ✓ 🚀 -- Index stores lower() of every email -- Query applies lower() once to constant
✅ Fix 2 — citext Extension
CREATE EXTENSION citext; -- Change column to case-insensitive type: ALTER TABLE users ALTER COLUMN email TYPE citext; -- Plain index now works for all cases: CREATE INDEX idx_email ON users (email); -- All of these use the index AND match: WHERE email = 'Alice@Example.com' ✓ WHERE email = 'ALICE@EXAMPLE.COM' ✓ WHERE email = 'alice@example.com' ✓
✅ Fix 3 — pg_trgm for LIKE
CREATE EXTENSION pg_trgm; -- GIN trigram index supports LIKE '%x%': CREATE INDEX idx_email_trgm ON users USING GIN (email gin_trgm_ops); -- Now these use the index: WHERE email ILIKE '%alice%' ✓ WHERE email LIKE '%example.com' ✓ -- Even middle-of-string patterns!
🕵️Why Planner Ignores Index
Function on column (lower(email)) = index can't be used. Function must be in the index, not the query column.
🛡️Enforce Consistency at Storage
Trigger or CHECK constraint to normalize email to lowercase on INSERT/UPDATE.
