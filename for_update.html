four variants — click each
FOR UPDATE
SELECT ... FOR UPDATE
Locks the row. Others block and wait until you commit. Most common variant.
SKIP LOCKED
FOR UPDATE SKIP LOCKED
Skip rows that are already locked. Get the next available one. Ideal for ticket/job queues.
NOWAIT
FOR UPDATE NOWAIT
If the row is locked, throw an error immediately. Don't wait at all.
FOR SHARE
SELECT ... FOR SHARE
Weaker lock. Others can also read with FOR SHARE. Only blocks FOR UPDATE writers.

the complete booking function
production-ready booking — click each section
BEGIN;
-- Grab one seat atomically. SKIP LOCKED means:
-- "if a seat is being booked right now, skip it"
SELECT id FROM seats
 WHERE event_id = 42 AND available = true
 LIMIT 1
 FOR UPDATE SKIP LOCKED;

-- If NULL → all seats locked or sold. Return "sold out".
IF seat_id IS NULL THEN ROLLBACK; RETURN 'sold out'; END IF;

UPDATE seats SET
  available = false,
  user_id = current_user_id,
  booked_at = NOW()
WHERE id = seat_id;

COMMIT; -- lock released, seat visible as booked to everyone

serializable — no locking, just dependency tracking
Serializable does not lock rows. All transactions run simultaneously and freely read/write. PostgreSQL secretly builds a dependency graph as they run. If it finds a cycle — meaning the result is impossible to explain as any sequential ordering — it aborts one transaction.

the decision flow
Q: Can I express the operation as a single atomic UPDATE?
  YES → Use atomic UPDATE. No isolation level needed.
  NO ↓
Q: Do I need to read a row, decide, then update that same row?
  YES → Use FOR UPDATE (or SKIP LOCKED for pools).
  NO ↓
Q: Do I need multiple queries to see a consistent snapshot?
  YES → Use REPEATABLE READ.
  NO ↓
Q: Could two transactions' decisions interact in a logically impossible way?
  YES → Use SERIALIZABLE + retry logic.
  NO → READ COMMITTED is fine.
