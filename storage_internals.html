👁️ MVCC in action — Two transactions see different versions
While transaction 10891 is still running (not yet committed), an older transaction 10500 still reads the old tuple. This is MVCC — readers don't block writers, writers don't block readers. Both versions coexist on the same page simultaneously.
Page 3 — What each transaction sees

Transaction 10891 sees:

Both old AND new tuples physically exist on the same page right now.
🧹 After Commit + VACUUM — Dead tuple reclaimed
Transaction 10891 commits. Now all transactions started after it will see 'Alicia'. The old tuple is now truly dead — nobody can ever see it again. But it still physically occupies space! VACUUM eventually comes along and marks that space as reusable for future inserts.

HOT Updates — Heap-Only Tuple Optimization
❌ Normal UPDATE (index update needed)
UPDATE users SET email = '...' — email IS indexed
1. Index (email) → old ctid (3,1)
2. Heap page: old tuple marked dead
3. New tuple written at (3,2)
4. Index must be updated → (3,2) ← expensive!
Index: email_idx → (3,1) [stale]
Must update: email_idx → (3,2) 💸
✅ HOT Update (no index update)
UPDATE users SET bio = '...' — bio is NOT indexed
1. Index (email) → item_ptr[1]
2. item_ptr[1]: old tuple marked dead
3. New tuple written on SAME page
4. item_ptr[1] chains to new tuple ✓
Index: email_idx → item_ptr[1]
item_ptr[1] → HOT chain → new tuple 🚀
HOT Chain — How item pointers forward to new tuple
Index → item_ptr[1] (offset=100) → old tuple (xmax=10891, HEAP_HOT_UPDATED flag) →
                                         ↓ HOT chain pointer
                                         new tuple (xmin=10891, HEAP_ONLY_TUPLE flag)
Requirement: New tuple must fit on the same page as the old tuple. If the page is full, HOT is not possible — a regular update with index maintenance is used instead.
📍 What Is an Offset? How ctid Works
The Core Idea — Byte Position From The Start
An offset is simply how many bytes to skip from the beginning of something to reach a target.
Think of a 8192-character book page where every character has a position number starting at 0.
Offset 200 means: "skip the first 200 bytes, your data starts here."

ctid = (page_number, item_pointer_number) — NOT a byte offset directly
(3, _)
Page Number
Which 8KB page block in the heap file. Skip page × 8192 bytes to reach it.
(_, 2)
Item Pointer Number
Which slot in the item pointer array. Each slot is 4 bytes and stores the actual byte offset of the tuple within the page.
💡 Why two levels? (Page → Item Ptr → Byte offset)
If indexes stored direct byte offsets, then every time VACUUM moved a tuple within a page to compact free space, every index pointing to that tuple would need updating — potentially hundreds of index entries.

With item pointer indirection: VACUUM moves the tuple, updates only the one item pointer, and all indexes (which store the item pointer number, not the byte offset) automatically find the new location. 🎯
The Book Analogy
📚 A real book page
char pos:
0    "Hello world, this is a "
24  "database" page..."
"database" is at offset 24 — skip 24 characters from the start of the page.
🗄️ A PostgreSQL page
byte 0   [page header]
byte 24  [item ptrs...]
byte 7950 [your tuple]
Your row is at byte offset 7950 — skip 7950 bytes from start of page 3.
❓ Why not just use row numbers?
Rows are variable size — offsets work for anything

No — VACUUM, CLUSTER, pg_repack can change it
