The visibility rule: xmin must be committed in your snapshot, AND xmax must not be committed in your snapshot.

the xmin / xmax rule in plain english
xmin — birth certificate
The transaction that INSERT-ed this row. A row only exists to you if xmin's transaction is in your "committed" list.
xmin=100 + txn 100 committed
→ row was born ✓ I can see it
xmax — death certificate
The transaction that DELETE-d or UPDATE-d this row. Row is dead to you only if xmax's transaction committed in your snapshot.
xmax=200 + txn 200 in-progress
→ not dead yet ✓ I still see it

what a snapshot actually contains
Your transaction 201's snapshot — taken when you ran BEGIN
xmin
= 198
Lowest XID still in progress. Any row with xmin < 198 and committed is definitely visible.
xmax
= 202
Next XID to be assigned. Any row with xmin ≥ 202 definitely doesn't exist yet for you.
xip
= [198, 200]
In-Progress list. XIDs between xmin and xmax that are STILL running. Their rows are invisible to you.

the key insight — why this replaces locks
❌ Without MVCC (locking)
Writer locks the row.
Reader must WAIT until writer commits or rolls back.
High concurrency = long queues.
Reader blocks writer, writer blocks reader.
✓ With MVCC
Writer creates a new row version.
Reader sees the old version via snapshot.
Both work simultaneously, no waiting.
Readers never block writers, writers never block readers.
four levels — click each to see the key difference
Read Committed
PostgreSQL default · most apps use this
New snapshot taken per statement. Each query sees all commits up to that moment.
BEGIN;
SELECT ... ← snapshot 1 taken here
-- other txn commits --
SELECT ... ← snapshot 2 taken here
COMMIT;
Repeatable Read
One snapshot for entire transaction
Snapshot taken once at BEGIN. All queries see the same frozen state.
BEGIN; ← snapshot taken ONCE here
SELECT ... ← uses same snapshot
-- other txn commits --
SELECT ... ← still same snapshot
COMMIT;
Serializable
Strongest · detects dependency cycles
Snapshot + conflict detection. Aborts transactions that would produce impossible results.
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT ... UPDATE ...
COMMIT;
-- or: ERROR: could not serialize
-- retry if aborted
Read Uncommitted
Theoretical · PostgreSQL ignores it
In theory: can read uncommitted data. In PostgreSQL: identical to Read Committed. Dirty reads are impossible in PostgreSQL by design.
SET TRANSACTION ISOLATION LEVEL
  READ UNCOMMITTED;
-- PostgreSQL silently treats this
-- as READ COMMITTED

four anomalies — what can go wrong
Dirty Read
Read uncommitted data from another transaction. If it rolls back, you read data that never officially existed.
PostgreSQL:
Never happens.
All levels safe.
Non-Repeatable
Read
Read the same row twice, get different values. Another transaction committed a change between your two reads.
Read Committed:
Can happen.
Repeatable Read+:
Prevented.
Phantom Read
Run same query twice, different rows appear. Another transaction inserted rows matching your WHERE clause between your two queries.
Read Committed:
Can happen.
Repeatable Read+:
Prevented (PG).
Lost Update
Two transactions read a value, both compute updates, both write back. Last write wins, first write is silently lost.
Read Committed:
Can happen.
Use atomic UPDATE
or FOR UPDATE.
Serialization
Anomaly
Each transaction is individually correct, but together they produce a result that could never happen if they ran one at a time.
All lower levels:
Can happen.
Serializable:
Prevented.
