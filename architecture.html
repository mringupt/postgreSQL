What it is: Any program connecting to PostgreSQL — your Node.js/Python/Java app, a GUI like DBeaver or TablePlus, or the psql command-line tool.

How it connects: Over TCP to port 5432 using the PostgreSQL wire protocol. ORMs like Prisma, SQLAlchemy, and ActiveRecord generate SQL and send it as plain text over this protocol.

What the client actually does: Almost nothing server-side. It sends an SQL string and waits. All parsing, planning, and execution happens inside Postgres. The client just receives rows back.

The hidden cost of each connection: Every connection causes postmaster to fork a new OS process consuming ~5–10MB of RAM just for process overhead, before any query runs. 500 open connections = 5GB RAM overhead before a single query runs.
Protocol: libpq / PostgreSQL wire protocol Each connection = one forked OS process (~5–10MB RAM) 500+ raw connections without pooling = serious memory pressure
What it is: The very first process that starts when you run PostgreSQL. It is the parent of everything else — all backend processes and background workers are its children.

What it does:
  • Listens on port 5432 for incoming TCP connections
  • Forks a new backend process for each connection via Unix fork() — creating a full OS-level copy of itself
  • Allocates shared memory at startup (shared_buffers, WAL buffers, lock tables) — all child processes inherit access to this shared region
  • Spawns and monitors all background workers; if one crashes, postmaster restarts it
  • Handles OS signals: SIGHUP = reload config, SIGTERM = graceful shutdown, SIGQUIT = immediate stop

If postmaster dies: Every connection drops instantly. All in-flight transactions are lost. Postgres is completely down. On restart it must do crash recovery via WAL replay.
☠ If postmaster dies, the entire cluster goes down instantly PID stored in $PGDATA/postmaster.pid
What it is: A dedicated OS process forked by postmaster for each client connection. Lives exactly as long as the connection is open, then exits.

The query pipeline inside every backend:
  • Parser — turns SQL text into a parse tree. Catches syntax errors.
  • Analyzer — resolves table/column names against the system catalog. Catches "relation does not exist" errors.
  • Rewriter — applies rules, expands views into their underlying SQL
  • Planner / Optimizer — generates all candidate execution plans, estimates cost using table statistics, picks the cheapest. This is where index vs seq scan decisions happen.
  • Executor — walks the plan tree, fetches 8KB pages from shared_buffers (or disk), applies filters, joins, aggregates, returns results

Why separate processes and not threads? Crash isolation. If one query crashes its backend, only that connection dies. Others are completely unaffected. Trade-off: ~5–10MB RAM overhead per connection.
✓ Crash in one backend cannot affect others 200+ connections without a pooler = RAM pressure
What it does: Continuously scans shared_buffers looking for dirty pages and writes them to heap files on disk — proactively, before a checkpoint forces it.

What exactly is a "dirty" page? When a backend modifies data in RAM (INSERT/UPDATE/DELETE), the 8KB page in shared_buffers becomes different from its copy on disk. That page is now "dirty" — like editing a Google Doc offline: your browser has the new version, the server does not. Until written back to disk, a crash could lose that change (WAL protects against this, but the heap page itself is still unsaved).

shared_buffers (RAM): [page A: DIRTY 🔴] modified, disk is stale [page B: clean ✅] identical to disk, safe [page C: DIRTY 🔴] modified, disk is stale | | bgwriter writes dirty pages to disk v disk (heap files): [page A: saved ✅] now matches RAM [page C: saved ✅] now matches RAM
Why bgwriter matters: Without it, ALL dirty flushing would happen at checkpoint time — causing a massive I/O spike every few minutes. bgwriter spreads writes smoothly over time, like doing homework daily instead of cramming the night before.

If bgwriter falls behind: Backend processes themselves flush pages when they need free buffer slots. You see buffers_backend_fsync rising in pg_stat_bgwriter — a warning sign.
Monitor: pg_stat_bgwriter.buffers_clean High buffers_backend_fsync = bgwriter not keeping up
What it does: Periodically forces ALL dirty pages in shared_buffers to be written to disk, then writes a "checkpoint record" into the WAL — creating a known-good point that crash recovery can start from.

Timeline: [checkpoint] ---WAL records--- [WAL records] ---[CRASH] | +-- On restart, replay WAL from HERE forward +-- Not from the very beginning of time +-- Recovery time = distance from last checkpoint to crash
Key settings:
  • checkpoint_timeout — max time between checkpoints (default 5 min)
  • max_wal_size — also triggers a checkpoint if WAL grows this large (default 1GB)
  • checkpoint_completion_target = 0.9 — spread checkpoint I/O over 90% of the interval to avoid I/O spikes at the end
Frequent checkpoints = constant I/O pressure Infrequent checkpoints = slow crash recovery (minutes) checkpoint_completion_target=0.9 smooths the I/O burst
What it does: Periodically scans tables for dead tuples — stale row versions left behind by UPDATE and DELETE — and reclaims that wasted space.

Why dead tuples exist (MVCC): PostgreSQL uses Multi-Version Concurrency Control. When you UPDATE a row, Postgres does NOT overwrite it in-place. It writes a brand new version and stamps the old one as "dead" (sets xmax). This lets concurrent transactions still see the old version. VACUUM physically removes these dead versions later.

UPDATE users SET name='Charlie' WHERE id=1; Heap page after update: [id=1, name=Alice, xmax=99] ← DEAD 🔴 (old version, marked for removal) [id=1, name=Charlie, xmin=99] ← LIVE ✅ (new version) autovacuum later removes the dead row: [id=1, name=Charlie, xmin=99] ← only live version remains [free space................ ] ← space reclaimed for new rows
Also does:
  • Runs ANALYZE to refresh query planner statistics
  • Freezes old transaction IDs to prevent XID wraparound — a catastrophic scenario where data becomes invisible (covered in Topic 5)
☠ NEVER disable autovacuum in production Monitor: pg_stat_user_tables.n_dead_tup n_dead_tup growing unchecked = table bloat, slow scans
What it does: Periodically flushes the in-memory WAL buffer to the WAL files on disk. Runs every wal_writer_delay (default 200ms).

How WAL works: Before ANY data page is modified, Postgres first writes a record describing the change to WAL. On a crash, even if dirty data pages were never flushed to heap files, Postgres replays WAL to reconstruct all committed changes. WAL is the foundation of durability and replication.

Transaction modifies a row: 1. Write WAL record → WAL buffer (RAM) ← instant 2. Modify data page → shared_buffers (RAM) ← instant 3. COMMIT: flush WAL buffer → disk ← sequential write (fast) client gets confirmation here 4. bgwriter writes dirty data page later ← async, can happen later Key: WAL is append-only (sequential). Sequential disk writes are 10–100x faster than random writes. This is why Postgres is both durable AND fast.
Durability settings:
  • synchronous_commit = on — wait for WAL flush before confirming commit. Safest. Default.
  • synchronous_commit = off — confirm immediately; WAL flushed async. Up to 200ms of commits can be lost on crash. Use only for non-critical data.
  • fsync = off — NEVER in production. Risks total data directory corruption on any crash.
WAL = foundation of crash safety AND replication fsync=off = risk of complete data corruption on crash
What it is: A fixed region of RAM allocated when Postgres starts. Holds 8KB data pages and is shared by ALL backend processes simultaneously. It is Postgres's own cache layer, separate from the OS page cache.

The two-layer cache system:
Backend needs to read a row: Step 1: Check shared_buffers → Found? Buffer HIT ✅ (no disk I/O, instant) → Not found? Go to Step 2 Step 2: Check OS page cache (kernel memory) → Found? Still no physical disk I/O! Load into shared_buffers. → Not found? Go to Step 3 Step 3: Read from physical disk 🔴 → load into shared_buffers This is the slow path. Avoid by having enough RAM.
Clean vs Dirty pages:
  • Clean page — identical to disk copy. Can be evicted from RAM anytime with no penalty.
  • Dirty page — modified in RAM, disk copy is stale. Must be flushed by bgwriter or checkpointer before it can be evicted.
Why 25% of RAM, not 100%? The OS independently caches disk reads. Giving Postgres all RAM leaves no room for the OS cache (Step 2 disappears). 25% Postgres + 75% OS = two independent cache layers.

What is OLTP? Online Transaction Processing — your live app traffic: user logins, order inserts, balance checks. Many small fast queries running concurrently. The same "hot" pages are read thousands of times per second. Cache hit rate matters enormously. Target 99%+.

SELECT sum(heap_blks_hit) / nullif(sum(heap_blks_hit)+sum(heap_blks_read),0) AS hit_rate FROM pg_statio_user_tables;
Set to 25% of total RAM Hit rate below 95% for OLTP = consider increasing shared_buffers Two cache layers: shared_buffers (Postgres) + OS page cache (kernel)
What it is: The maximum RAM for a single sort or hash join operation node. Critical: per operation node — not per query, not per connection. Per individual node in the query plan tree.

The multiplication trap — why this gets dangerous fast:
Your query plan has 4 nodes: Hash Join ← gets its own work_mem (64MB) Sort ← gets its own work_mem (64MB) Hash Aggregate ← gets its own work_mem (64MB) Sort ← gets its own work_mem (64MB) 1 query = 4 nodes × 64MB = 256MB 4 workers × 256MB = 1,024MB (1GB for ONE query!) 100 connections × 1GB = 100GB 🔥 → Server runs out of RAM → Linux OOM Killer activates → Kills Postgres (biggest process) instantly → Database crashes. All connections drop. No warning.
What is OOM? Out Of Memory. When RAM is exhausted, the Linux OOM Killer instantly kills the biggest process (usually Postgres) with no warning, no log, no graceful shutdown.

Too small? Sorts spill to disk temp files: Sort Method: external merge Disk: 48MB — 10–100x slower.

What is OLTP? Online Transaction Processing — live app traffic with many short concurrent queries. Set work_mem small globally; increase per-session for heavy one-off queries.
-- postgresql.conf (safe global for OLTP) work_mem = 8MB -- Per-session for a heavy analytics query SET work_mem = '512MB'; SELECT ... GROUP BY ... ORDER BY ...; RESET work_mem; -- back to 8MB -- Safe sizing formula: -- (RAM × 0.25) / (max_connections × avg_plan_nodes) -- 32GB RAM, 200 connections, 3 nodes avg: -- 8192 / 600 = ~13MB → round down to 8MB
Too high = OOM kill = instant database crash, no warning Too low = disk spill = 10–100x slower sorts Set small globally; bump per-session for analytics
OOM = Out Of Memory. When the server's RAM is completely exhausted, the Linux kernel activates its last-resort mechanism: the OOM Killer.

1. A process requests more RAM from the OS 2. OS has none available — all memory is allocated 3. OOM Killer activates: scans all running processes 4. Assigns each a score (roughly: RAM used × time) 5. Sends SIGKILL to the highest-scoring process 6. SIGKILL = instant death. No cleanup. No log. No warning. 7. Postgres (often the biggest process) gets killed. 8. All connections drop simultaneously.
What happens to Postgres:
  • postmaster dies instantly — no graceful shutdown
  • All active client connections drop simultaneously
  • All in-flight transactions are lost (WAL ensures clean rollback on recovery)
  • On restart: crash recovery via WAL replay — seconds to minutes depending on checkpoint gap
How to confirm OOM killed Postgres:
$ dmesg | grep -i "oom" Out of memory: Kill process 1234 (postgres) score 847 ... Killed process 1234 (postgres), total-vm:8388608kB ... # Postgres logs just show abrupt stop — process was killed # before it could write an error message.
Main causes in Postgres:
  • work_mem too high globally — most common. Multiplies across all connections and plan nodes.
  • Too many connections — 500 × 10MB overhead = 5GB before any queries run
  • Runaway parallel query — many workers each using large work_mem
  • maintenance_work_mem too high — used during VACUUM and CREATE INDEX
☠ No warning, no log, no graceful shutdown — instant death Prevention: conservative work_mem + PgBouncer connection pooling Monitor RAM: alert at 85% usage before it becomes critical

Transaction modifies data: 1. WAL record → WAL buffer (RAM) ← instant 2. Data page → shared_buffers (RAM) ← instant 3. COMMIT: WAL buffer flushed to disk ← sequential write (fast) client gets confirmation 4. bgwriter writes data page later ← async Why WAL writes are fast despite touching disk: WAL is append-only — always sequential I/O. Sequential disk writes are 10–100x faster than random writes. This is why Postgres is both durable AND performant.
Size: Auto-set to 1/32 of shared_buffers, capped at 16MB. Rarely needs manual tuning.

When to increase: If backends are waiting on WAL buffer locks (visible in pg_stat_activity), set explicitly to 16MB or 32MB in postgresql.conf.
Default auto-tuning is fine for most workloads Sequential appends = much faster than random data page writes
What it is: A number you tell the query planner to assume is available for caching — combining shared_buffers AND the OS page cache. It allocates absolutely zero RAM. It is purely a hint to the planner.

How the planner uses it: When choosing between an index scan and a sequential scan, the planner estimates how many pages will be in memory vs on disk. A larger effective_cache_size makes index scans look more attractive.

Scenario: 64GB RAM server, large 10GB table WRONG: effective_cache_size = 128MB (Postgres default) Planner thinks: "almost nothing is cached" Planner picks: Sequential Scan (assumes disk I/O for everything) Reality: Data IS cached — index scan would be instant Result: Unnecessarily slow full-table scan CORRECT: effective_cache_size = 48GB (75% of 64GB) Planner thinks: "most data is probably in memory" Planner picks: Index Scan Result: Fast query, correct decision
Recommended value: ~75% of total RAM. On a 64GB server: effective_cache_size = 48GB

Most common mistake: Leaving it at the default 128MB on a server with 32GB+ RAM. The planner runs with completely wrong assumptions and picks sequential scans everywhere.
Default of 128MB is almost always dangerously low — change this Set to ~75% of total RAM Zero RAM allocated — purely a hint to the query planner
Why "heap"? In computer science, a heap is storage with no particular order — things land wherever there is space, with no sorting. PostgreSQL heap files are exactly that: rows are stored in no sorted order. A new row goes into the first available free slot. No attempt to keep rows ordered by ID or anything else.

Heap file — completely unordered on disk: Page 1: [id=5, Alice] [id=1, Bob] [id=9, Carol] [id=3, Dan] Page 2: [id=7, Eve] [DEAD 🔴] [id=8, Frank] [free space] Page 3: [id=4, Grace] [id=6, Henry] [free space....................] INSERT id=10 → goes into first free slot (Page 2, after dead tuple) No attempt to place id=10 near id=9. Heap does not care about order.
Structure of each 8KB page:
  • Page header (24 bytes) — LSN, checksum, free space pointers
  • Item pointers — array of (offset, length) pairs, one per tuple
  • Free space — in the middle, shrinks as rows are added
  • Tuples (rows) — packed upward from the bottom of the page
The dirty page connection: When a backend modifies a row, that 8KB heap page in shared_buffers changes while the disk copy stays the same — the page is now "dirty." bgwriter and checkpointer eventually flush it back to the heap file on disk.
Disk heap page: [Alice, Bob] ← CLEAN ✅ (matches RAM) RAM shared_buf: [Alice, Charlie] ← DIRTY 🔴 (Bob→Charlie, not saved yet) | | bgwriter flushes to disk v Disk heap page: [Alice, Charlie] ← CLEAN ✅ again
Why UPDATE never overwrites in-place: MVCC requires old row versions to remain visible to concurrent transactions. UPDATE writes a NEW tuple and stamps the old one dead (xmax). autovacuum later removes dead tuples.
8KB page = the fundamental unit of all I/O in Postgres Dead tuples = bloat. autovacuum reclaims them. Unordered = fast inserts, but full scans need indexes
What it is: An append-only sequential log of every change made to the database, written to disk BEFORE the actual data pages are modified. The "write-ahead" guarantee: WAL record always comes first.

Three things WAL enables:
  • Crash recovery: On restart, Postgres reads the last checkpoint record then replays WAL forward. Even if dirty data pages were never flushed to heap files, all committed changes are recovered. No committed data is ever lost.
  • Streaming replication: Standbys receive the same WAL stream and apply it in order — staying byte-for-byte identical to the primary. Physical replication is just WAL shipping.
  • Point-in-time recovery (PITR): Archive WAL files continuously. With a base backup + WAL archive, you can restore to any exact moment in the past.
$PGDATA/pg_wal/ on disk: 000000010000000000000001 (16MB segment, complete) 000000010000000000000002 (16MB segment, complete) 000000010000000000000003 (current, being appended) Each record: LSN | Transaction ID | Operation | Changed data Always sequential appends — never random writes → very fast
Critical rule: fsync = on must never be disabled. Without it, WAL sits in OS buffer cache and can be lost on power failure — leaving the database in a corrupt state with no recovery path.
Sequential append-only I/O = very fast fsync=off = silent data corruption risk on any crash or power loss wal_level=replica required for streaming replication
What it is: Separate files on disk storing a data structure that maps key values to physical row locations (ctid = page number + offset within page) in the heap file. An index stores pointers to rows, not the row data itself.

B-Tree index on users(email): "alice@x.com" → ctid=(page 4, offset 2) [sorted] "bob@x.com" → ctid=(page 1, offset 5) [sorted] "carol@x.com" → ctid=(page 9, offset 1) [sorted] Query: SELECT * FROM users WHERE email = 'bob@x.com' With index: 1. B-Tree traversal → ctid = (page 1, offset 5) [fast, log(n)] 2. Heap fetch: read page 1, get tuple at offset 5 [1 page read] Total: 2 steps Without index: Read ALL pages of the heap file looking for bob@x.com Could be thousands of pages on a large table
Index types in PostgreSQL:
  • B-Tree (default) — balanced sorted tree. Handles =, <, >, BETWEEN, ORDER BY, LIKE 'prefix%'. Use for almost everything.
  • GIN — for JSONB, arrays, full-text search. Stores per-element entries.
  • GiST — geometric types, range types. Extensible, supports overlap/contain operators.
  • BRIN — tiny index storing min/max per block range. Only useful for naturally ordered append-only data (time-series).
  • Hash — equality-only O(1) lookup. Rarely preferred over B-Tree in practice.
Why Postgres sometimes ignores your index: An index scan involves random I/O (jump to heap page at ctid). If the query returns 20% of the table, that is thousands of random I/Os — often worse than one sequential scan through the whole heap. The planner calculates this and picks correctly.
Index-Only Scan = no heap fetch needed (fastest possible path) Index → ctid → heap page (two-step lookup) Planner may correctly choose Seq Scan if selectivity is too low
